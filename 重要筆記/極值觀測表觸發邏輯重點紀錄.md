# 極值觀測表 (dgObserver) 觸發邏輯與優化重點紀錄

這份文件記錄了極值觀測表（`dgObserver`）的運作邏輯、核心計算指標、日誌報表差異，以及最新實作的**「兩層極值篩選漏斗」效能優化機制**。

---

## 一、 核心觸發機制 (A點與B點定義)

極值觀測表的核心精神是在市場發生**「局部極端價格」**時，觀測其**「速度」與「成交量」**的變化，藉此捕捉潛在的**「反轉點」**。

所有計算與狀態判定主要封裝在 `TradingEngine.CalcSimulationResultsInternal` 與 `TradingEngine.GetDurationsFull` 中。

### 1. A 點（觀察起點）之物理定義
系統會持續掃描 Tick 行情，尋找潛在的轉折點（A 點），分為兩種路徑：
* **抓新高反轉 (做空)**：
  * **往前看 (Pre)**：蒐集 `N` 筆同盤別的**「外盤」**成交紀錄。如果往前找的過程中，有價格高於 A 點，代表 A 點不是局部最高點，立刻捨棄。
  * **往後看 (Post)**：蒐集 `N` 筆對立盤別的**「內盤」**成交紀錄。如果往後找的過程中，價格又突破了 A 點，代表反轉失敗，立刻捨棄。
* **抓新低反轉 (做多)**：
  * 邏輯完全對稱。往前找 `N` 筆**「內盤」**（不能有更低價），往後找 `N` 筆**「外盤」**（不能有更低價）。

### 2. 觸發條件：量增且速增
當成功收集滿前段 (`Pre`) 與後段 (`Post`) 各 `N` 筆 Tick 時（即程式碼中的 `actPre >= currentN && actPost >= currentN`），系統會計算並比對兩個關鍵數值來決定是否真正觸發：
1. **平均速度 (`Avg`)**：
   * 將相鄰兩筆 Tick 的時間差加總後取平均。
   * 條件：`post.Value < pre.Value`。即**「後段的平均時間間隔，必須小於前段」**（反轉後的行情速度變快）。
2. **累積成交量 (`Vol`)**：
   * 將這 `N` 筆 Tick 的成交口數加總。
   * 條件：`preVol < postVol`。即**「反方向的成交量，必須大於原方向」**（反轉後力道放大）。

### 3. B 點（確認達標點）與預設防守價 (StopLoss)
* **確認點 (B 點)**：當後段 (`Post`) 收集到第 `N` 筆 Tick 的瞬間，系統視為「確認達標」。這一筆 Tick 的價格與時間就會被記錄為 `TrigPrice` 與 `TrigTime`。
* **防守停損價 (StopLossPrice)**：
  * 做空（抓新高）時，防守價預設綁定為 A 點價格（`PrevHigh`）。
  * 做多（抓新低）時，防守價預設綁定為 A 點價格（`PrevLow`）。

---

## 二、 階段三：時序停損與破價狀態機

觸發條件成立後，系統會按時間順序，掃描 B 點之後的所有最新 Tick 進行防守監控：
* **做空模式**：若後續有任何一筆 Tick 的價格 **大於 (`>`)** 當時的 A 點防守價。
* **做多模式**：若後續有任何一筆 Tick 的價格 **小於 (`<`)** 當時的 A 點防守價。

一旦發生上述情況，系統判定防守失敗，程式會將該筆觀測標記為 `IsBroken = true`（已破價），並記錄 `BreakTime`。此時 UI 的 `StopLossDisplay` 欄位會從數字變更為顯示 `{價位}(已破)`。

## 三、 A 點篩選之效能優化 (兩層極值漏斗)

為解決大數據量下頻繁執行 `GetDurationsFull` 進行複雜盤別統計與時間差計算帶來的 CPU 負載，系統引入了兩層**必要條件過濾器**。這兩層過濾器與原來的動態 $N$ 計算完全相容，且**保證最終訊號結果 100% 邏輯等價**。

### 1. 漏斗架構與原理解析
1. **第一關：相鄰極值過濾 (Adjacent Extreme Filter) — 粗篩網**
   * **邏輯**：A 點的價格必須大於或等於（做空）或小於或等於（做多）緊鄰其左邊與右邊 the Tick。
     * 做空：`Price[idx] >= Price[idx-1]` 且 `Price[idx] >= Price[idx+1]`。
   * **物理意義**：直接排除非「局部凸點/凹點」的趨勢中繼點。若隔壁的 Tick 就已超越 A 點，後續向左向右掃描時必定會在一開始就宣告破價失敗。
   * **效能貢獻**：僅需進行 $O(1)$ 的數值比較，**直接過濾掉約 70% 處於單邊趨勢中的無效 Tick**。
2. **第二關：動態 N 區間過濾 (Dynamic N Range Filter) — 細篩網**
   * **邏輯**：直接使用當前 Tick 算出的動態 $N$ 值（`currentN`），掃描左右各 `currentN` 筆 Tick。
     * 做空：左右各 `currentN` 筆 Tick 中不得有任何價格大於 A 點。
   * **物理意義**：若在最近的 `currentN` 筆 Tick 內就已經存在超越 A 點的價格，那麼在往左/往右集滿 $N$ 筆同向 Tick 之前，必定會撞上該價格而 `break` 淘汰。
   * **效能貢獻**：在第一關過濾後剩下的 30% 候選點中，利用快速的區間極值掃描，進一步**淘汰掉約 20%（佔總數）的微小局部轉折點**。
3. **漏斗效應結果**：
   * 原本 100% 的 Tick 都需要執行沉重的盤別分類與間隔時間差計算。
   * 經過雙層漏斗後，**僅剩約 10% 的真正極值候選點需要進入 `GetDurationsFull` 計算**，CPU 效能大幅優化。

### 2. 程式碼修改對照
```csharp
// 以做空路徑為例：
int currentN = useDynamicN && dynamicNMap != null ? dynamicNMap[aIdx] : obsN;

// 第一關：相鄰極值過濾 (左右各 1 Tick)
if (aIdx > 0 && aPrice < trades[aIdx - 1].Price) continue;
if (aIdx < totalTradesCount - 1 && aPrice < trades[aIdx + 1].Price) continue;

// 第二關：動態 N 區間過濾 (左右各 currentN Tick)
bool isInvalid = false;
int checkLeftLimit = Math.Max(0, aIdx - currentN);
for (int i = aIdx - 1; i >= checkLeftLimit; i--)
{
    if (trades[i].Price > aPrice) { isInvalid = true; break; }
}
if (!isInvalid)
{
    int checkRightLimit = Math.Min(totalTradesCount - 1, aIdx + currentN);
    for (int i = aIdx + 1; i <= checkRightLimit; i++)
    {
        if (trades[i].Price > aPrice) { isInvalid = true; break; }
    }
}
if (isInvalid) continue;

// 通過雙層漏斗後，才執行複雜的指標統計
var (pre, preVol, post, postVol, threshold, trigTime, ... ) = GetDurationsFull(...);
```

## 四、 極值觀測表 vs. 日誌表單之 A 點篩選核心差異

除了觀測筆數 $N$ 的來源不同外（觀測表支援動態 $N$；日誌表單使用固定 $N$），兩者在計算與去重機制上有本質上的不同：

| 比較項目 | 極值觀測表 (`dgObserver`) | 日誌表單 (離線回測報表) |
| :--- | :--- | :--- |
| **遍歷機制** | **跳躍式**<br>當 A 點成功觸發 B 點後，搜尋起點會直接**跳至 B 點的下一個 Tick**，跳過 A-B 之間的所有 Tick。 | **連續式**<br>對整段歷史 Tick 進行無跳躍的連續掃描，不會因為觸發而跳過任何區間。 |
| **去重機制** | **局部動態去重**<br>僅過濾與「上一次成功觸發相同價格」的 A 點。若中間有觸發其他價格，相同價格之後仍可再次觸發。 | **全域首現去重**<br>使用全域 `seenKeys` 進行去重。在整段時段中，同方向的同一個價格**只保留首次出現的 A 點**，後續相同價格全部捨棄。 |
| **未達標處理** | **不保留**<br>僅呈現已達標或當前正在觀測中的潛在訊號。 | **完整保留**<br>「曾未達標」的極值點也會保留於報表中（其停損價顯示為 `N/A`）。 |
| **前置篩選門檻** | **無前置門檻**<br>直接依據 $N$ 筆同向掃描結果判定。 | **有嚴格的前置門檻**<br>Tick 必須先符合創高/低、突破或時差 $\ge 30$ 秒等 UI 極限條件。 |
